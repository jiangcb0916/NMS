# Sangfor NMS Next 全站 UI 重构 — 结构化需求文档

## 1. 需求背景

- **业务目标**：将当前分阶段演进形成的多套页面风格统一为一套专业、克制、高密度的网络运维工作台，提升值守人员扫描状态、定位异常和执行操作的效率。
- **预期价值**：减少模块切换时的视觉割裂，统一常用操作位置和状态表达，提高桌面值守效率，并确保移动端具备基础可用性。
- **目标用户**：内网网络运维人员、系统管理员和需要查看网络运行状态的值守人员。

## 3. 功能详述

### F-01：全局深色工作台外壳

**Requirement**：系统必须使用统一的深色运维外壳；桌面端必须提供可折叠紧凑侧栏并记忆状态，移动端必须默认收起导航。

**Scenario 1：桌面日常值守**
- Given：用户在桌面浏览器登录系统
- When：用户展开或折叠侧栏并切换模块
- Then：内容区应稳定调整宽度，当前菜单清晰可见，后续访问保留侧栏偏好

**Scenario 2：移动端访问**
- Given：视口宽度为手机尺寸
- When：用户打开导航并选择模块
- Then：导航以抽屉方式显示，选择后自动收起，页面不产生横向溢出

### F-02：统一视觉与组件规范

**Requirement**：系统必须统一基础颜色、文字层级、间距、边框、圆角、按钮、表单、标签、提示、弹窗和表格样式。

**Scenario 1：跨模块操作**
- Given：用户在不同模块使用搜索、筛选、刷新、分页和弹窗
- When：用户执行相同类型操作
- Then：控件位置、状态反馈和视觉含义应保持一致

### F-03：模块信息架构优化

**Requirement**：每个页面必须按“页面状态栏、关键摘要、操作区、主体内容”的顺序组织；允许合并重复标题和重排现有信息，但不得删除现有业务能力。

**Scenario 1：进入业务模块**
- Given：用户切换至任一功能模块
- When：页面加载完成
- Then：用户应优先看到模块名称、数据来源或更新时间、核心状态和主要操作

### F-04：监控页面统一

**Requirement**：运行总览、防火墙、流量分析、OSDWAN、网络拓扑、交换机和无线页面必须采用一致的监控语言，同时根据业务内容保留图表、拓扑和状态列表的差异。

**Scenario 1：异常扫描**
- Given：某接口异常、未配置或返回空数据
- When：监控页面完成请求
- Then：对应区域应明确显示异常、未配置或暂无数据，不得出现空白区域或拖垮其他模块

### F-05：管理页面统一

**Requirement**：设备、客户端和用户管理页面必须使用高密度表格，并统一摘要、工具栏、筛选、分页、行操作和弹窗。

**Scenario 1：搜索和分页**
- Given：列表包含多页数据
- When：用户搜索、筛选、排序或切换分页
- Then：操作状态、总数和当前显示数量应清晰，自动刷新不得清空用户输入或重置分页

### F-06：状态与可访问性交互

**Requirement**：所有异步区域必须覆盖加载、空数据、错误和保留旧数据状态；可操作控件必须具备可见焦点和可读名称。

**Scenario 1：请求失败**
- Given：页面已有一份有效数据
- When：刷新请求失败
- Then：页面应保留上一份有效数据并显示失败提示，不得直接清空主体内容

**Scenario 2：键盘操作**
- Given：用户使用键盘导航
- When：用户按 Tab 或 Escape
- Then：焦点应清晰可见，Escape 可关闭弹窗和临时菜单，焦点返回触发控件

### F-07：响应式适配

**Requirement**：桌面端必须保持高信息密度；移动端必须将指标、工具栏和详情改为纵向或双列布局，复杂表格应提供行卡片或局部滚动兜底。

### F-08：登录体验统一

**Requirement**：登录页必须采用与工作台一致的品牌、色彩和反馈规范，不得更改现有登录、跳转和错误处理逻辑。

## 4. 业务规则与约束

1. 保持 Flask、Jinja、原生 JavaScript 和 CSS 技术栈，不引入新的前端框架或构建流程。
2. 保持现有后端 API、数据库、鉴权和外部系统逻辑不变。
3. 不删除现有搜索、筛选、分页、刷新、导入导出、CRUD、图表、拓扑和检测能力。
4. 视觉采用中性黑灰；绿色表示正常、黄色表示提醒、红色表示异常、蓝色表示速率或检测路径。
5. 禁止装饰性渐变、强发光、英文装饰标题、营销式大标题和无业务价值的指标卡。
6. 面板使用细边框和小圆角，紧凑但不得造成文字重叠、截断失真或误触。
7. 用户已有的未跟踪 `specs/` 目录不属于本次改动范围。
8. 本阶段仅开发和本地验证，不自动发布或部署。

## 6. 验收标准

1. 登录页以及 10 个主要业务视图使用同一套深色工作台规范，无明显旧样式残留或模块间割裂。
2. `1920x1080` 和 `1280x720` 下侧栏可折叠，常用状态与操作无需大范围滚动即可看到。
3. `390x844` 下导航可用，工具栏合理换行，页面无整体横向滚动，主要操作可点击。
4. 搜索、筛选、刷新、分页、导入导出、弹窗、图表、拓扑和 CRUD 行为保持可用。
5. 页面明确覆盖加载中、暂无数据、未配置、接口异常和保留旧数据状态。
6. Tab 导航具有可见焦点，Escape 可关闭弹窗和临时菜单，状态提示具备可读语义。
7. 不新增前端运行依赖，不修改 API 契约、数据库结构、鉴权和外部系统配置。
8. `node --check static/js/app.js`、Python 语法检查、`scripts/smoke_test.py` 和 `git diff --check` 通过。
9. 浏览器逐项验证运行总览、防火墙、流量、OSDWAN、拓扑、交换机、无线、设备、客户端、用户和登录页。
